# UX требования: старт Mini App, выбор роли, Личный кабинет

## Цель
Сделать поведение Mini App понятным и предсказуемым:
- при первом входе нового пользователя показывать выбор роли и шаги первичной настройки;
- для уже зарегистрированных пользователей сразу показывать функционал по их роли (без “выбора роли”);
- “Аккаунт” открывается только по явному нажатию кнопки в углу.

## Контекст / проблема (как сейчас ощущается)
1. При открытии приложения у пользователей сразу открывается экран “Аккаунт”. Это воспринимается как тупо/неуместно.
2. Есть более правильная схема экранов:
   - новый пользователь -> экран выбора роли;
   - зарегистрированный -> функционал по роли;
   - доступ к “Аккаунту” только через кнопку в углу.

---

## Блок 1. Определение “нового” пользователя
Нужно однозначно решить, что считать “новым/не настроенным”.

Предполагаемое правило (уточнить при первой проверке фронта):
- `user.role === 'draft'` ИЛИ
- у роли нет завершенного профиля (например, `ownerProfile` отсутствует для owner или `workerProfile` отсутствует для worker).

### Требование
Если пользователь “не настроен”:
- показываем экран `RoleSelect` (выбор работодателя/работника);
- после выбора роль переводим в активную стадию настройки профиля.

Если пользователь “настроен”:
- экран выбора роли не показываем вообще;
- сразу редиректим на функционал по роли.

---

## Блок 2. Стартовая маршрутизация (главное правило)
### Требование
На первом рендере Mini App:
1. Определить `user` через `GET /users/me?telegramId=...`.
2. Проверить, “настроен” ли профиль по роли.
3. Показать нужную главную страницу:
   - `owner` -> `OwnerHome` (кнопки/функционал поиска/запроса сотрудников);
   - `worker` -> `WorkerHome` (статистика/выходы на смены/отклики).
4. При этом:
   - экран “Аккаунт” НЕ открывать автоматически;
   - “Аккаунт” открывать только при явном переходе по кнопке.

---

## Блок 3. Роль-ориентированный функционал
### OwnerHome (работодатель)
Набор функций (MVP):
- кнопка/раздел “Запросить работника на ПВЗ” (выбор ПВЗ, создание смены/публикация);
- список/таблица откликов по активным сменам;
- действие “выбрать одного” (назначение).

### WorkerHome (работник)
Набор функций (MVP):
- блок “Статистика” по выходам на смены (или список последних смен);
- блок “Мои отклики” (откликнуться/снять отклик);
- при необходимости — отображение уведомлений о назначении/занято.

---

## Блок 4. Кнопка “Аккаунт” (в углу)
### Требование
- Кнопка “Аккаунт” всегда видна пользователю (в углу интерфейса).
- При клике открывает `AccountScreen`.
- По прямой логике UX “Аккаунт” не должен быть стартовым экраном.

---

## Блок 5. Экран “Аккаунт”
### Что доступно
1. Просмотр текущей роли и профиля.
2. Изменение данных аккаунта/профиля.
3. Удаление аккаунта.
4. Смена роли:
   - при смене роли аккаунт с предыдущей ролью должен быть удален/зачищен так, чтобы старые зависимости не мешали.

### Требование к backend логике
Backend уже должен обеспечивать:
- при `switchRole` чистку зависимых данных старой роли;
- при `deleteAccount` проверку зависимостей (нельзя удалить, если есть связанные данные).

Фронту нужно лишь:
- вызвать соответствующие API;
- после смены роли показать корректный экран первичной настройки нового профиля.

---

## Блок 6. Явное открытие “Аккаунта” vs автозапуск
### Требование
Экран “Аккаунт” должен открываться только:
- когда пользователь нажал кнопку “Аккаунт”,
- либо когда бот/другой триггер явно запросил `settings=account`.

Если Mini App запускается “обычно”:
- никакие параметры, которые могут приводить к показу кабинета, не должны применяться автоматически.

---

## Блок 7. Что проверить при реализации (чеклист)
1. Новый пользователь -> стартует на `RoleSelect`, не на `AccountScreen`.
2. Пользователь owner -> стартует на `OwnerHome`.
3. Пользователь worker -> стартует на `WorkerHome`.
4. Кнопка “Аккаунт” всегда доступна и открывает `AccountScreen`.
5. “Смена роли” приводит к очистке данных прошлой роли (проверить после вызова `switchRole`).

---

## Блок 8. Примечания (где будем сверяться)
- Фронт: `apps/mini-app/src/*`
- API: `apps/api/src/modules/*` и эндпоинт `GET /users/me`, `POST /users/switch-role`, `DELETE /users/me`
